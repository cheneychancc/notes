面试官问 JSONP 的局限性，本质是在考察你对**浏览器同源策略（SOP）**的理解，以及为什么在现代开发中它几乎被 **CORS** 完全取代了。

可以从以下四个核心维度来回答：

---

### 1. 仅支持 GET 请求（最大的痛点）
这是 JSONP 协议原理决定的。
* **原理**：JSONP 是通过动态创建 `<script>` 标签，利用其不受同源策略限制的特性来加载数据的。
* **局限**：`<script>` 标签只能发起 **GET** 请求。因此，你无法使用 POST 提交大数据，也无法使用 PUT、DELETE 等符合 RESTful 规范的动作。

### 2. 安全性问题 (XSS 风险)
JSONP 极其容易受到 **XSS（跨站脚本攻击）** 的威胁。
* **信任风险**：你执行的是第三方服务器返回的 JS 代码。如果该服务器被黑客攻陷，或者该接口被恶意注入，黑客可以返回一段盗取 Cookie 或监听键盘的恶意代码，你的浏览器会直接执行它。
* **缺乏防御机制**：它不像 CORS 那样可以精确控制哪些域名、哪些 Header 允许访问。

### 3. 错误处理机制薄弱
* **状态码不可知**：如果 JSONP 请求失败（比如 404 或 500 错误），`<script>` 标签不会返回 HTTP 状态码。
* **异常捕获难**：你很难区分是网络超时、服务端崩溃还是数据格式错误。通常只能依靠设置一个定时器，如果规定时间内回调函数没被调用，就认为请求失败。

### 4. 无法处理复杂的交互
* **无法设置 Header**：因为是利用标签加载，你无法在请求中自定义 `Authorization` 或 `Content-Type` 等请求头。
* **Cookie 受限**：虽然可以带上 Cookie，但相比现代 Fetch API 的配置（如 `credentials: 'include'`），它的灵活性极差。

---

### 💡 深度对比：为什么现在都用 CORS？

| 特性 | JSONP | CORS (现代方案) |
| :--- | :--- | :--- |
| **请求类型** | 仅 GET | **所有类型 (GET/POST/PUT/...)** |
| **安全性** | 差（容易被 XSS） | **高（由服务端配置白名单）** |
| **错误处理** | 无法获取状态码 | 完善的错误捕获机制 |
| **底层原理** | HTML 标签漏洞 | **HTTP 协议标准** |

---

### 🎤 面试加分回答建议

> “JSONP 曾是跨域问题的‘救命稻草’，但它本质上是一个利用浏览器安全漏洞的 **Hack 手段**。
>
> 在我的项目中，我们全面转向了 **CORS**。一方面是因为 JSONP **无法支持 POST 请求**，这对于数据标注平台这种需要频繁提交大体积 JSON 数据的场景是不可能的；另一方面，CORS 提供了基于 **HTTP 响应头**的精确安全控制，能有效防御 CSRF 和 XSS 攻击。
>
> 只有在需要兼容 **IE 极低版本**或对接某些非常古老的第三方系统时，我们才会考虑这种已经被时代淘汰的技术。”
