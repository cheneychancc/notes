要理解 React Fiber，最直白的解释是：**它是 React 的一套“可中断、可恢复”的更新调度机制。**

在 Fiber 出现之前（React 15 及以前），React 的更新是**同步且不可中断**的。一旦开始渲染，浏览器就会被死死卡住，直到整个组件树渲染完成。

---

### 1. 为什么需要 Fiber？（解决“卡顿”问题）

想象一下，如果你有一个包含 1000 个节点的列表需要更新：
* **旧版（Stack Reconciler）**：React 会像递归一样一气呵成地处理完。如果这个过程耗时 100ms，而显示器每 16.7ms (60fps) 就需要刷新一次，那么用户就会感到明显的**丢帧和卡顿**。
* **新版（Fiber）**：React 把大任务拆解成很多个“微小的工作单元”。每处理完一个单元，React 都会停下来问浏览器：“你现在忙吗？有用户点击或者动画要处理吗？”如果有，就先让给浏览器，等浏览器空闲了再继续。



---

### 2. Fiber 的三层含义

#### ① 作为架构：从“堆栈”到“链表”
旧版渲染依赖函数调用栈（Stack），一旦开始就没法停。
Fiber 改用了**动态的工作循环**。它把组件树转换成了一个**双向链表**结构。每个 Fiber 节点都有 `child`（第一个子节点）、`sibling`（下一个兄弟节点）和 `return`（父节点）指针。
*这种链表结构让 React 可以随时跳出循环，以后再顺着指针找回来。*



#### ② 作为静态数据结构：一个普通的 JS 对象
每个组件实例都会对应一个 Fiber 对象，里面存储了：
* 组件类型、DOM 节点。
* 待处理的 `props` 和 `state`。
* **副作用（Side Effects）**：比如需要做的 DOM 插入、删除或更新。

#### ③ 作为工作单元：
Fiber 还是一个执行单元，React 会在 `requestIdleCallback`（类似的思想）的时间片内执行它。

---

### 3. Fiber 的双缓存机制 (Double Buffering)

这是 Fiber 性能稳定的核心。React 内存中同时存在两棵 Fiber 树：
1.  **current 树**：当前屏幕上显示的树。
2.  **workInProgress 树**：正在内存中构建的树。

当 `setState` 触发更新时，React 在 `workInProgress` 树上进行计算。计算完成了，只需要把根节点的指针指向新树，瞬间完成替换。这就像**显卡渲染图像**：先在后缓冲区画好，然后一次性拷贝到前缓冲区。

---

### 4. 渲染的两个阶段 (Phases)

为了实现可中断，Fiber 把渲染分成了两个阶段：

* **第一阶段：Render 阶段（可中断）**
    * React 遍历 Fiber 树，找出哪些地方需要变动。
    * 这个阶段不会有任何真实的 DOM 操作。
    * 因为是纯计算，所以可以被打断、重启、甚至废弃。

* **第二阶段：Commit 阶段（不可中断）**
    * React 根据第一阶段算出的结果，执行真实的 DOM 操作（增删改）。
    * 为了保证 UI 变化的一致性，这个阶段必须**同步执行**，不能停。

---

### 💡 面试加分回答

> “Fiber 的本质是 React 模拟了**操作系统的‘时间片分时调度’**。它通过将同步渲染改为**异步并发渲染**，解决了 JavaScript 主线程长任务阻塞 UI 响应的问题。
>
> 它的核心实现依赖于**链表结构**（取代了递归栈）和**双缓存技术**。这使得 React 可以实现 Concurrent Mode（并发模式），支持像 `useTransition` 和 `useDeferredValue` 这样高级的性能优化手段。”

**你之前在写 3D 可视化页面时，有没有遇到过因为 JS 计算量太大（比如解析几十万个点云数据）导致 React 按钮点不动的情况？那其实就是典型的“由于缺乏时间片分时调度”导致的阻塞。**
